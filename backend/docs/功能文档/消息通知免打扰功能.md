# 消息通知免打扰功能

## 功能概述

消息通知免打扰功能允许用户自定义各类消息的通知偏好，开启免打扰后，该类型的新消息将自动标记为已读，不再产生提醒。

## 功能特性

### 1. 支持的消息类型

- **评论通知**（message_type=1）：当有人评论您的内容时
- **回复通知**（message_type=2）：当有人回复您的评论时
- **点赞通知**（message_type=3）：当有人点赞您的内容或评论时

**注意**：系统通知（message_type=0）和审核通知（message_type=4）不可设置免打扰，确保重要系统消息和审核结果能够及时送达。

### 2. 自动已读机制

当用户对某类消息设置免打扰后：

1. **新消息自动已读**：该类型的新消息创建时会自动标记为已读
2. **历史消息处理**：设置免打扰时，会将该类型的所有历史未读消息标记为已读
3. **懒创建优化**：对于群发消息（按角色、按部门、系统通知），只有在用户查看时才创建中间表记录

### 3. 用户界面

#### 免打扰设置对话框

- 位置：消息通知页面（`/#/user/messages`）顶部"免打扰设置"按钮
- 功能：统一管理所有消息类型的免打扰状态
- 交互：每个消息类型对应一个开关按钮，点击"确定"批量保存更改

#### 设置项说明

每个消息类型包含：
- **图标和名称**：直观展示消息类型
- **描述文字**：说明该类型消息的触发场景
- **开关按钮**：控制免打扰状态
- **状态提示**：显示当前是否已开启免打扰

## 技术实现

### 数据模型

```python
class UserNotificationPreference(CoreModel):
    """用户通知偏好设置"""
    user = models.ForeignKey(to='Users', ...)  # 用户
    message_type = models.IntegerField(...)     # 消息类型
    is_muted = models.BooleanField(...)         # 是否免打扰
    muted_at = models.DateTimeField(...)        # 设置免打扰时间
```

### API 接口

#### 1. 获取用户偏好设置

```
GET /api/system/user_notification_preference/get_preferences/
```

返回示例：
```json
{
  "code": 2000,
  "data": {
    "1": {"is_muted": false, "muted_at": null},
    "2": {"is_muted": false, "muted_at": null},
    "3": {"is_muted": true, "muted_at": "2026-03-05T10:30:00Z"},
    "4": {"is_muted": false, "muted_at": null}
  }
}
```

#### 2. 设置免打扰

```
POST /api/system/user_notification_preference/set_mute/
Content-Type: application/json

{
  "message_type": 3
}
```

功能：
- 创建或更新用户偏好记录
- 将该类型的所有历史未读消息标记为已读
- 返回已标记消息数量

#### 3. 取消免打扰

```
POST /api/system/user_notification_preference/cancel_mute/
Content-Type: application/json

{
  "message_type": 3
}
```

功能：
- 更新用户偏好记录，设置 `is_muted=False`
- 后续新消息将正常提醒

### 消息创建流程

#### 指定用户消息（target_type=0）

```python
# 1. 查询用户的免打扰偏好
muted_users = UserNotificationPreference.objects.filter(
    user_id__in=users,
    message_type=message_type,
    is_muted=True
).values_list('user_id', flat=True)

# 注意：系统通知(0)和审核通知(4)不检查免打扰状态
if message_type not in [0, 4]:
    # 2. 创建中间表记录时，根据免打扰状态设置 is_read
    for uid in users:
        is_read = uid in muted_users  # 免打扰用户自动已读
        MessageCenterTargetUser.objects.create(
            users_id=uid,
            messagecenter_id=message_id,
            is_read=is_read
        )

    # 3. 只向未免打扰的用户推送 WebSocket 通知
    for uid in users:
        if uid not in muted_users:
            websocket_push(uid, message)
```

#### 群发消息（target_type=1/2/3）

群发消息采用懒创建策略：
- 创建时不写入中间表
- 用户查看时才创建记录
- 创建时检查免打扰状态，自动标记为已读

#### 批量保存逻辑

```typescript
const handleSaveMuteSettings = async () => {
  // 找出有变化的设置项
  const changedItems = muteSettings.value.filter(item => item.changed);
  
  // 批量调用 API
  const promises = changedItems.map(item => {
    const url = item.muted
      ? '/api/system/user_notification_preference/set_mute/'
      : '/api/system/user_notification_preference/cancel_mute/';
    return request({ url, method: 'post', data: { message_type: item.type } });
  });
  
  await Promise.all(promises);
};
```

## 使用场景

### 场景 1：减少点赞通知干扰

用户发布了热门内容，收到大量点赞通知，影响正常使用：

1. 打开消息通知页面
2. 点击"免打扰设置"按钮
3. 开启"点赞通知"免打扰
4. 点击"确定"保存

结果：
- 所有历史点赞消息标记为已读
- 新的点赞消息自动已读，不再提醒
- 其他类型消息正常接收

### 场景 2：临时关闭评论通知

用户参与了热门讨论，短时间内收到大量回复：

1. 开启"回复通知"免打扰
2. 讨论结束后，关闭免打扰
3. 恢复正常接收回复通知

### 场景 3：审核通知无法免打扰

用户想要关闭审核通知，但发现无法设置：

原因：
- 审核通知属于系统重要通知
- 与系统通知一样，不可设置免打扰
- 确保用户能及时了解内容审核结果

## 注意事项

1. **系统通知和审核通知不可免打扰**：确保重要系统消息和审核结果能够送达
2. **免打扰是自动已读**：不是屏蔽消息，消息仍会创建，只是自动标记为已读
3. **历史消息处理**：设置免打扰时会将该类型的所有历史未读消息标记为已读
4. **WebSocket 优化**：免打扰用户不会收到 WebSocket 实时推送，减少服务器负载
5. **数据库优化**：使用唯一索引 `(user, message_type)` 防止重复记录
